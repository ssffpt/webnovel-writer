# CLI流程 vs Web流程：差异对比

## 一、总体定位差异

| 维度 | CLI原始设计 | 当前Web实现 |
|------|-----------|------------|
| **定位** | 完整的创作系统，覆盖从创意到成稿全流程 | 以查看/编辑为主的文件浏览器 + 轻量AI助手 |
| **核心理念** | 防幻觉三定律（大纲即法律/设定即物理/发明需识别），双Agent架构 | 无此概念，用户自由编辑任意文件 |
| **流程保障** | 强制6步顺序执行，禁止并步/跳步 | 用户随时做任何事，无流程约束 |

---

## 二、各命令流程差异

### 1. `/webnovel-init` — 项目初始化

| 维度 | CLI | Web |
|------|-----|-----|
| **入口** | `/webnovel-init` 命令 | CreateWizard 3步向导 |
| **采集步骤** | **6步渐进式采集**：故事核→角色骨架→金手指→世界观→创意约束包→一致性复述 | **3步简化表单**：基本信息→主角设定(可选)→确认 |
| **交互深度** | 每步都有引导性问题，AI辅助收集，可给候选方向 | 纯表单填写，无AI引导 |
| **核心差异** | **创意约束包**（Step 5）完全缺失：反套路库映射→生成2-3套创意包→三问筛选→五维评分 | 没有创意约束机制，仅有"核心卖点"文本框 |
| **核心差异** | **角色骨架**仅采主角名，缺少：欲望、缺陷、结构、感情线配置、反派分层 | 仅有主角名+金手指 |
| **核心差异** | **金手指**缺少：风格、可见度、不可逆代价、成长节奏、条件必收项（系统流/重生/传承的额外字段） | 仅有名称+类型 |
| **核心差异** | **世界观**完全缺失：世界规模、力量体系、势力格局、社会阶层 | 无此步骤 |
| **核心差异** | **一致性复述**完全缺失：输出摘要草案→用户确认→才执行生成 | 确认页仅展示表单数据 |
| **充分性闸门** | 6项必须全部通过才可执行：书名/题材/规模/主角欲望+缺陷/世界+力量/创意约束 | 无闸门机制，表单提交即执行 |
| **执行方式** | `webnovel.py init` 带30+参数 | `POST /api/project/create` 带~8个参数 |
| **后处理** | Patch总纲（补齐故事一句话/核心暗线/创意约束/反派分层/爽点里程碑）、写idea_bank.json | 仅创建基础目录结构 |

**影响**：初始化质量大幅降低，缺少设定基线，后续plan/write无法获得足够的上下文约束。

---

### 2. `/webnovel-plan` — 卷级规划

| 维度 | CLI | Web |
|------|-----|-----|
| **入口** | `/webnovel-plan [卷号]` | OutlinePage点击"生成第N卷大纲"按钮 |
| **执行步骤** | **8步完整流程**：加载数据→构建设定基线→选择卷→节拍表→时间线表→卷骨架→章节大纲(分批16字段)→回写设定集 | 按钮触发`POST /api/tasks`，action.type=`plan_outline` |
| **实际执行** | 全部8步由AI Agent逐步执行 | **claude_runner仅做preflight+文件存在性校验**，不做实际规划！ |
| **节拍表** | 必须生成：中段反转（必填）、危机链（至少3次）、卷末新钩子 | 未实现 |
| **时间线表** | 必须生成：时间基准、时间跨度、关键倒计时 | 未实现 |
| **卷骨架** | Strand规划+爽点密度+伏笔+约束触发 | 未实现 |
| **章节大纲** | 每章16个必填字段（目标/阻力/代价/时间锚点/Strand/爽点/钩子等） | 未实现 |
| **回写设定集** | 新增事实写回设定集，冲突标记BLOCKER | 未实现 |
| **7项验证** | 爽点密度/Strand比例/总纲一致性/约束频率/完整性/时间线/设定补全 | 未实现 |

**影响**：plan完全未实现，大纲页面只是手动编辑markdown文件的文本编辑器。

---

### 3. `/webnovel-write` — 章节创作（最核心差异）

| 维度 | CLI | Web |
|------|-----|-----|
| **入口** | `/webnovel-write [章号]` | ChapterPage编辑textarea + AI助手说"帮我写" |
| **完整流程** | **6步强制顺序**：Context Agent→正文起草→风格适配→审查→润色→Data Agent | **无流程**，用户直接在textarea写或AI助手触发任务 |
| **Context Agent** | 生成7板块任务书+Context Contract全字段+写作执行包 | 未实现 |
| **正文起草** | 从创作执行包出发，遵循大纲约束，2000-2500字 | 用户手写，或AI助手触发后claude_runner仅做preflight+extract-context，**不生成正文** |
| **风格适配** | Step 2B定向改写模板腔/说明腔/机械腔 | 未实现 |
| **六维审查** | 6个Checker并行执行（核心3+条件3），必须由Task子代理执行，结果落库 | 未实现 |
| **润色** | 修复critical→high→medium/low，Anti-AI全文终检 | 未实现 |
| **Data Agent** | 9个子步骤（实体提取→消歧→写入state/index→摘要→场景切片→RAG向量→风格样本→债务） | 未实现 |
| **模式支持** | 标准/--fast/--minimal | 无 |
| **流程硬约束** | 禁止并步/跳步/自审替代/自创模式 | 无约束 |
| **Git备份** | Step 6自动提交 | 未实现 |
| **断点恢复** | workflow start-task/complete-task 记录 | claude_runner调用了workflow start-task但仅best-effort |

**影响**：这是最核心的差距。当前Web的"写章节"实际上只是手动编辑文本文件，完全缺失了防幻觉的整套保障机制。

## 三、架构层面差异

### 1. 双Agent架构缺失

| CLI | Web |
|-----|-----|
| Context Agent（读）：写作前构建创作任务书 | 无 |
| Data Agent（写）：写后提取实体/更新状态/建RAG索引 | 无 |

### 2. RAG向量检索缺失

| CLI | Web |
|-----|-----|
| 写作时通过RAG检索相关设定/伏笔/前文 | 无RAG，所有"智能"靠关键词匹配的chat API |

### 3. 六维并行审查缺失

| CLI | Web |
|-----|-----|
| 6个Checker并行执行，结果落库 | 无审查机制 |

### 4. 工作流状态机缺失

| CLI | Web |
|-----|-----|
| 明确的Step定义和状态转换，断点可恢复 | TaskService仅有pending→running→completed/failed，无Step粒度 |

### 5. claude_runner是"半真实命令映射"

代码注释明确说明（`claude_runner.py:21-23`）：

> 仓库当前并未暴露可直接非交互调用的 `/webnovel-write` / `/webnovel-plan` / `/webnovel-review` 等完整技能入口，因此这里先实现"半真实命令映射"

实际只做了preflight校验和extract-context提取，没有真正执行任何skill的完整流程。

---

## 四、总结：核心差距清单

按优先级排列：

1. **write流程完全缺失** — 没有Context Agent/起草/审查/润色/Data Agent的6步流程
2. **plan流程完全缺失** — 没有节拍表/时间线/卷骨架/章节大纲生成
3. **init严重缩水** — 6步→3步，缺少角色骨架/世界观/创意约束包/一致性复述
4. **review流程完全缺失** — 没有六维审查和报告
5. **双Agent架构缺失** — Context Agent和Data Agent均未实现
6. **RAG检索缺失** — 无法解决长篇写作的遗忘问题
7. **工作流状态机缺失** — 无Step粒度的断点和恢复
8. **query/resume缺失** — 伏笔紧急度/节奏分析/中断恢复均未实现

简而言之：当前Web版是一个**文件编辑器+轻量AI助手**，而CLI是一个**完整的AI驱动创作系统**。Web版需要将CLI的每个skill流程映射为Web交互，保持流程约束和防幻觉机制不变，只是把命令行交互变成Web页面交互。

## 五、CLI各Skill完整流程参考

### `/webnovel-init` 完整6步

1. **Step 1 故事核与商业定位**：书名/题材(复合)/目标规模/一句话故事/核心冲突/目标读者
2. **Step 2 角色骨架与关系冲突**：主角姓名/欲望/缺陷/结构/感情线/反派分层
3. **Step 3 金手指与兑现机制**：类型/名称/风格/可见度/不可逆代价/成长节奏 + 条件必收项
4. **Step 4 世界观与力量规则**：世界规模/力量体系/势力格局/社会阶层 + 题材相关项
5. **Step 5 创意约束包**：反套路库映射→2-3套创意包→三问筛选→五维评分→用户选择
6. **Step 6 一致性复述与最终确认**：摘要草案→用户确认→执行生成→Patch总纲→写idea_bank

### `/webnovel-plan` 完整8步

1. **Step 1**：加载项目数据（state.json/总纲/设定集/idea_bank）
2. **Step 2**：构建设定基线（增量补齐，不清空重写）
3. **Step 3**：选择卷（确认卷名/章节范围/特殊需求）
4. **Step 4**：生成卷节拍表（中段反转/危机链/卷末钩子）
5. **Step 4.5**：生成卷时间线表（时间基准/跨度/倒计时）
6. **Step 5**：生成卷骨架（Strand规划/爽点密度/伏笔/约束触发）
7. **Step 6**：生成章节大纲（每章16字段，分批生成）
8. **Step 7**：回写设定集（新增事实写回，冲突标记BLOCKER）
9. **Step 8**：验证+保存+更新状态（7项验证检查）

### `/webnovel-write` 完整6步

1. **Step 1 Context Agent**：7板块任务书+Context Contract+写作执行包
2. **Step 2A 正文起草**：遵循大纲约束，2000-2500字，禁止占位符
3. **Step 2B 风格适配**（--fast/--minimal跳过）：定向改写模板腔/说明腔/机械腔
4. **Step 3 审查**：6维并行Checker（核心3+条件3），必须由Task子代理执行，结果落库
5. **Step 4 润色**：修复critical→high→medium/low，Anti-AI全文终检
6. **Step 5 Data Agent**：实体提取→消歧→写入state/index→摘要→场景切片→RAG向量→风格样本→债务
7. **Step 6 Git备份**：自动提交

### `/webnovel-review` 完整8步

1. **Step 1**：加载参考（core-constraints等）
2. **Step 2**：加载项目状态
3. **Step 3**：并行调用检查员（Task子代理，6维）
4. **Step 4**：生成审查报告（6维评分+修改优先级+改进建议）
5. **Step 5**：保存审查指标到index.db
6. **Step 6**：写回审查记录到state.json
7. **Step 7**：处理关键问题（critical必须用户决策）
8. **Step 8**：收尾

### `/webnovel-query` 6种查询类型

1. **标准查询**：角色/境界/金手指等，搜索设定集
2. **伏笔分析**：三层分类(核心3.0x/支线2.0x/装饰1.0x)+紧急度公式
3. **金手指状态**：基本信息/等级/技能/升级条件/发展建议
4. **节奏分析**：Strand连续/断档检测，Quest>5/Fire>10/Constellation>15警告
5. **标签格式**：实体标签规范查询
6. **紧急债务**：伏笔回收紧急度排序

### `/webnovel-resume` 恢复流程

1. 加载恢复协议（禁止智能续写）
2. 加载数据规范
3. 确认上下文充足
4. 检测中断状态（workflow detect）
5. 展示恢复选项（命令/中断时间/已完成步骤/剩余步骤/选项风险）
6. 执行恢复（删除重来/Git回滚）
7. 继续任务（可选）
