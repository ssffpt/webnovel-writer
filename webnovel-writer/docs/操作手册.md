# Webnovel Writer 操作手册

## 一、项目概述

Webnovel Writer 是一套基于 Claude Code 的交互式网文创作系统。通过一系列结构化的 Skill（技能命令），实现从项目初始化、大纲规划、章节写作、质量审查到数据管理的完整创作闭环。

### 核心理念

| 定律 | 说明 |
|------|------|
| **大纲即法律** | 遵循大纲，不擅自发挥 |
| **设定即物理** | 遵守设定，不自相矛盾 |
| **发明需识别** | 新实体必须入库管理 |

### 系统架构

```
┌─────────────────────────────────────────────────┐
│                  Claude Code                     │
├─────────────────────────────────────────────────┤
│  Skills (8个): init / plan / write / review /    │
│               query / resume / dashboard / learn │
├─────────────────────────────────────────────────┤
│  Agents: Context Agent / Data Agent / 6维 Checker│
├─────────────────────────────────────────────────┤
│  Data: state.json / index.db / vectors.db        │
└─────────────────────────────────────────────────┘
```

### Strand Weave 节奏系统

| Strand | 含义 | 理想占比 |
|--------|------|---------|
| **Quest** | 主线剧情 | 60% |
| **Fire** | 感情线 | 20% |
| **Constellation** | 世界观扩展 | 20% |

节奏红线：Quest 连续不超过 5 章，Fire 断档不超过 10 章，Constellation 断档不超过 15 章。

## 二、核心概念详解

本系统涉及大量网文创作领域的专业术语。以下对每个核心概念做详细解释，帮助你理解系统的设计逻辑。

### 2.1 金手指（Golden Finger）

**什么是金手指？**

"金手指"是网文术语，指主角拥有的**超越常人的特殊能力或优势**，是主角区别于普通角色的核心竞争力。这个词源自游戏修改器（Cheat Engine），意为"开挂"。

**金手指的作用**：
- 让主角有资格参与更高层次的冲突
- 为爽点（打脸、逆袭）提供能力基础
- 控制剧情节奏（通过金手指的升级/限制制造张力）

**金手指的类型举例**：

| 类型 | 说明 | 典型例子 |
|------|------|---------|
| 系统流 | 获得一个游戏化系统 | 签到系统、任务系统、抽奖系统 |
| 重生/穿越 | 带着记忆回到过去或穿越到异世界 | 重生回高考前、穿越成反派 |
| 传承/器灵 | 获得上古强者传承或灵器 | 戒指里的老爷爷、神秘玉佩 |
| 特殊体质 | 天生拥有特殊身体能力 | 万毒不侵体、先天剑体 |
| 无金手指 | 主角无特殊外挂，靠智谋/努力 | 智斗流、苟道流 |

**本系统要求**：
- 每个项目必须明确金手指类型（可以是"无金手指"）
- 金手指**必须有代价**（如寿命消耗、副作用、使用次数限制），或明确说明"无代价"的理由
- 金手指的设计要服务于核心卖点，不能喧宾夺主

---

### 2.2 题材（Genre）

**什么是题材？**

题材是小说的**世界观背景和故事类型**，决定了读者群体和创作规范。

**系统内置 37+ 题材模板**，分为四大类：

| 分类 | 包含题材 | 特点 |
|------|---------|------|
| 玄幻修仙类 | 修仙、系统流、高武、西幻、无限流、末世、科幻 | 力量体系明确，升级打怪为主 |
| 都市现代类 | 都市异能、都市日常、都市脑洞、现实题材、电竞、直播文 | 现代背景，贴近生活 |
| 言情类 | 古言、宫斗宅斗、青春甜宠、豪门总裁、职场婚恋、种田等 | 感情线为主线 |
| 特殊题材 | 规则怪谈、悬疑脑洞、历史古代、知乎短篇、克苏鲁 | 独特的世界观规则 |

**复合题材**：支持 `题材A+题材B` 组合（如 `都市脑洞+规则怪谈`），建议主辅比例 7:3。

---

### 2.3 爽点（Cool Point）

**什么是爽点？**

爽点是让读者产生**"爽感"**的情节设计，是网文吸引读者持续追读的核心手段。

**爽点的 8 种模式**：

| 模式 | 说明 | 典型场景 |
|------|------|---------|
| 装逼打脸 | 嘲讽→反转→震惊 | 被嘲笑后展示真实实力 |
| 扮猪吃虎 | 示弱→暴露→碾压 | 隐藏身份的高手出手 |
| 越级反杀 | 差距→策略→逆转 | 弱者以弱胜强 |
| 打脸权威 | 权威→挑战→成功 | 挑战宗门长老并获胜 |
| 反派翻车 | 得意→反杀→落幕 | 反派嚣张后被打脸 |
| 甜蜜超预期 | 期待→超预期→升华 | 意料之外的甜蜜互动 |
| 迪化误解 | 随意行为→配角脑补→读者优越感 | 主角随便说句话，别人以为是深意 |
| 身份掉马 | 伪装→关键时刻→揭露→震惊 | 隐藏身份被揭穿 |

**爽点密度标准**：
- 常规章节：1-2 个小爽点
- 关键章节：2-3 个爽点，至少 1 个中等爽点
- 高潮章节：3-4 个爽点，至少 1 个大爽点

---

### 2.4 钩子（Hook）

**什么是钩子？**

钩子是章节末尾或章节内设置的**悬念或期待**，目的是让读者产生"想知道接下来怎样"的冲动，驱动他们继续阅读下一章。

**6 种钩子类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 危机钩 | 危险逼近，读者想知道主角如何应对 | "门外传来脚步声，越来越近" |
| 悬念钩 | 信息缺口，读者想知道真相 | "他死了三次，每次都记得" |
| 渴望钩 | 好事即将发生，读者期待兑现 | "明日宗门大比，所有人都将见证他的崛起" |
| 情绪钩 | 触发强烈情绪（愤怒/心疼/心动） | "她终于明白，那个男人一直在保护她" |
| 选择钩 | 两难抉择，读者想知道角色怎么选 | "救妻子还是救世界，他选择了第三条路" |
| 承诺钩 | 预告奖励或转折 | "三天后，他将迎来人生最大的转折" |

**钩子强度**：
- **strong**：卷末/关键转折，读者必须立刻知道后续
- **medium**：普通剧情章，读者想知道但可以等
- **weak**：过渡章，维持阅读惯性即可

---

### 2.5 伏笔（Foreshadowing）

**什么是伏笔？**

伏笔是作者在前期章节中**埋下的线索或暗示**，在后续章节中揭示或回收，给读者"原来如此"的满足感。

**伏笔三层分类**：

| 层级 | 说明 | 权重 | 示例 |
|------|------|------|------|
| 核心伏笔 | 与主线剧情直接相关 | 3.0x | 主角身世之谜、终极反派的真正目的 |
| 支线伏笔 | 与配角/支线相关 | 2.0x | 配角的隐藏身份、支线宝物的用途 |
| 装饰伏笔 | 氛围/细节类 | 1.0x | 某个不起眼的路人、某句随口说的话 |

**紧急度判定**：
- **紧急**：超过目标章数未回收，或核心伏笔超过 20 章未推进
- **警告**：超过 80% 目标章数，或支线伏笔超过 30 章
- **正常**：在计划范围内

---

### 2.6 Strand Weave（三线编织）

**什么是 Strand Weave？**

Strand Weave 是本系统的**节奏控制机制**，将小说情节分为三条线（Strand）交织推进，避免节奏单调。

| Strand | 含义 | 理想占比 | 典型内容 |
|--------|------|---------|---------|
| **Quest（主线）** | 核心任务与冲突推进 | 55-65% | 升级、战斗、夺宝、复仇 |
| **Fire（感情线）** | 情感关系发展 | 20-30% | 相识、暧昧、告白、师徒情 |
| **Constellation（世界观线）** | 扩展设定与背景揭示 | 10-20% | 新势力登场、身世揭秘、世界观展开 |

**节奏红线（必须遵守）**：
- Quest 连续不超过 **5 章**（否则读者疲劳）
- Fire 断档不超过 **10 章**（否则感情线断裂）
- Constellation 断档不超过 **15 章**（否则世界观单薄）

**前 30 章织网示例**：
```
第 1-5 章:   Quest（开局快速推进主线）
第 6 章:     Fire（首次与女主相遇）
第 7-10 章:  Quest（宗门大比）
第 11 章:    Fire（英雄救美）
第 12-14 章: Quest（秘境夺宝）
第 15 章:    Constellation（揭示新势力存在）
...
```

### 2.7 反套路（Anti-Trope）

**什么是反套路？**

反套路是**故意打破该题材的常见模式**，让故事有新鲜感和差异化。

**每个项目必须选择至少 1 条反套路规则**，例如：

| 规则 | 效果 |
|------|------|
| 金手指有代价且不可逆 | 每次使用都是抉择，增加紧张感 |
| 主角无法直接打脸，只能智斗 | 用策略替代武力，更有深度 |
| 反派与主角共享同一能力 | 镜像对抗，冲突更有层次 |
| 世界规则对主角不利 | 主角没有特权，更公平 |

**反套路的作用**：避免"烂大街"的套路，让作品在同题材中有差异化竞争力。

---

### 2.8 硬约束（Hard Constraints）

**什么是硬约束？**

硬约束是**不可违反的创作规则**，在整部小说中必须始终遵守。它们逼出独特的剧情解法。

**硬约束类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 世界观约束 | 世界运行的不可违背规则 | "修仙者不能杀凡人" |
| 能力约束 | 主角能力的硬性限制 | "金手指每天只能用一次" |
| 行为约束 | 主角行为的禁区 | "主角不能主动攻击" |
| 信息约束 | 主角知道/不知道的边界 | "主角不知道自己是穿越者" |
| 资源约束 | 稀缺资源的硬性限制 | "整个世界只有 100 颗灵石" |

---

### 2.9 核心卖点（Core Selling Point）

**什么是核心卖点？**

核心卖点是**读者选择这本书的唯一理由**，用一句话概括作品的核心吸引力。

**核心卖点公式**：
```
[主角身份/优势] + [核心行动] + [独特目标] = 核心卖点
```

**示例**：
- 重生商战大佬（身份）+ 回到高考前（行动）+ 改写人生巅峰（目标）
- 签到系统宿主（优势）+ 每日获得随机奖励（行动）+ 成为最强者（目标）

**核心卖点 vs 其他概念**：

| 概念 | 定义 | 区别 |
|------|------|------|
| 核心卖点 | 作品最核心的吸引力 | 唯一、贯穿全文、不可替代 |
| 题材 | 故事背景类型 | 范围广泛 |
| 金手指 | 主角特殊优势 | 服务于卖点的工具 |
| 爽点 | 读者爽感来源 | 卖点的具体表现形式 |

---

### 2.10 镜像对抗（Antagonist Mirror）

**什么是镜像对抗？**

镜像对抗是**反派设计的核心原则**：反派与主角共享同一"欲望或缺陷"，但采取相反的道路。

**示例**：
```
主角：林天，渴望力量保护家人
反派：慕容战，渴望力量统治一切

共享特质：对力量的渴望
主角道路：通过修炼和牺牲获得力量
反派道路：通过吞噬他人获得力量

→ 两人的对决不仅是实力对决，更是"获取力量的方式"之争
```

**镜像模板**：

| 共享特质 | 主角道路 | 反派道路 |
|---------|---------|---------|
| 渴望力量 | 通过努力/牺牲获得 | 通过掠夺/背叛获得 |
| 恐惧失去 | 学会放手/接受 | 控制/占有一切 |
| 追求认可 | 证明自己的价值 | 强迫他人认可 |

---

### 2.11 微兑现（Micro-Payoff）

**什么是微兑现？**

微兑现是章节内给读者的**"小收获"**，让读者感觉"这章没白看"。与大爽点不同，微兑现更轻量、更频繁。

**微兑现类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 信息兑现 | 揭示新信息/线索 | "原来那把钥匙的真正用途是..." |
| 关系兑现 | 关系推进/确认 | "她第一次主动握住了他的手" |
| 能力兑现 | 能力提升/新技能 | "他终于掌握了这门功法的精髓" |
| 资源兑现 | 获得物品/资源 | "储物袋里竟然还藏着一颗聚气丹" |
| 认可兑现 | 获得认可/面子 | "在场所有人看他的眼神都变了" |
| 情绪兑现 | 情绪释放/共鸣 | "他终于说出了压在心底的那句话" |

**每章建议**：至少 1 个微兑现，过渡章也不能"整章无收获"。

---

### 2.12 创意约束包（Creativity Constraints）

**什么是创意约束包？**

创意约束包是 `/webnovel-init` Step 5 产出的**差异化设计方案**，包含一组结构化的约束规则，确保你的作品不是"又一本套路文"。

**创意包包含**：
- **一句话卖点**：10 秒电梯演讲
- **反套路规则**：至少 1 条打破常规的规则
- **硬约束**：2-3 条不可违反的创作规则
- **主角缺陷驱动**：主角的缺陷如何制造冲突
- **反派镜像**：反派与主角的镜像关系
- **开篇钩子**：一句话钩子 + 开场场景 + 第一章末悬念

**五维评分系统**（满分 50 分）：

| 维度 | 权重 | 说明 |
|------|------|------|
| 新颖度 | 25% | 是否有创新，还是烂大街 |
| 市场性 | 20% | 目标受众大小 |
| 可写性 | 20% | 是否容易驾驭 |
| 爽点密度 | 20% | 爽感是否充足 |
| 长线潜力 | 15% | 能否支撑长篇连载 |

及格线 30 分，推荐线 40 分。

---

### 2.13 六维审查（6-Dimension Review）

**什么是六维审查？**

六维审查是本系统的**质量保障机制**，由 6 个专门的检查员（Checker）并行审查章节质量。

| 审查维度 | 检查内容 | 违反后果 |
|---------|---------|---------|
| 设定一致性 | 战力/地点/时间线是否矛盾 | 严重（破坏世界观） |
| 叙事连贯性 | 场景切换/时间推进是否合理 | 中等（影响阅读体验） |
| OOC 检查 | 人物行为是否偏离人设 | 中等（破坏角色可信度） |
| 追读力 | 钩子强度/期待管理是否到位 | 中等（影响留存） |
| 爽点密度 | 爽点数量和质量是否达标 | 轻微（影响阅读快感） |
| 节奏控制 | Strand 比例是否平衡 | 轻微（影响节奏感） |

---

### 2.14 Context Agent 与 Data Agent

**Context Agent（读 Agent）**

在写作**之前**运行，负责构建"创作任务书"。它会：
- 读取大纲、设定集、上一章内容
- 分析本章的目标、冲突、承接关系
- 生成完整的写作执行包（包含不可变事实清单、禁止事项、终检清单）

**Data Agent（写 Agent）**

在写作**之后**运行，负责数据闭环。它会：
- 从正文中自动提取新实体（角色/地点/物品/势力）
- 更新 state.json 和 index.db
- 生成章节摘要
- 建立 RAG 向量索引（用于后续检索）

**两者的关系**：Context Agent 为写作提供上下文，Data Agent 从写作结果中提取数据。形成"读→写→回写"的闭环。

---

### 2.15 RAG 检索

**什么是 RAG？**

RAG（Retrieval-Augmented Generation）是**检索增强生成**的缩写。在本系统中，它用于从已写的章节中检索相关信息，辅助后续章节的创作。

**工作原理**：
```
查询 → 向量检索 / 关键词检索 / 混合检索 → 融合排序 → Top-K 结果
```

**用途**：当写作新章节时，系统可以通过 RAG 检索之前章节中的相关描写、设定、对话，确保前后一致。

**配置**（可选）：
```bash
EMBED_API_KEY=your_embed_api_key    # 语义检索
RERANK_API_KEY=your_rerank_api_key  # 结果重排序
```

未配置时回退到关键词检索（BM25），仍可正常工作。

## 三、环境准备

### 前置要求

- Claude Code（已安装并配置）
- Python 3.11+
- 本项目插件已通过 Marketplace 安装

### 目录结构说明

系统涉及 4 层目录概念：

1. **WORKSPACE_ROOT**：Claude Code 工作区根目录（`${CLAUDE_PROJECT_DIR}`）
2. **WORKSPACE_ROOT/.claude/**：工作区级配置与指针
3. **PROJECT_ROOT**：真实小说项目根目录（由 `/webnovel-init` 按书名创建）
4. **CLAUDE_PLUGIN_ROOT**：插件缓存目录（不在项目内）

#### 小说项目目录结构

```
project-root/
├── .webnovel/            # 运行时数据（state/index/vectors/summaries）
├── 正文/                  # 正文章节
├── 大纲/                  # 总纲与卷纲
└── 设定集/                # 世界观、角色、力量体系
```

#### 工作区目录结构

```
workspace-root/
├── .claude/
│   ├── .webnovel-current-project   # 指向当前小说项目根
│   └── settings.json
├── 小说A/
├── 小说B/
└── ...
```

## 四、完整操作流程

从零开始创作一部小说的标准流程如下：

```
/webnovel-init → /webnovel-plan → /webnovel-write（循环）→ /webnovel-review（定期）
```

辅助命令：`/webnovel-query`、`/webnovel-resume`、`/webnovel-dashboard`、`/webnovel-learn`

### 第一步：项目初始化 `/webnovel-init`

**用途**：通过分阶段交互收集完整创作信息，生成项目骨架与约束文件。

**产出文件**：
- `.webnovel/state.json` — 项目状态文件
- `设定集/` — 世界观、力量体系、角色卡、金手指设计等
- `大纲/总纲.md` — 全书总纲
- `.webnovel/idea_bank.json` — 创意约束包

**交互流程（6 步）**：

#### Step 0：预检与上下文加载
- 确认当前目录可写
- 解析脚本目录并确认入口存在
- 加载最小参考文件

#### Step 1：故事核与商业定位
收集信息：
- 书名（可先给工作名）
- 题材（支持 A+B 复合题材）
- 目标规模（总字数或总章数）
- 一句话故事
- 核心冲突
- 目标读者/平台

**题材集合**：
- 玄幻修仙类：修仙、系统流、高武、西幻、无限流、末世、科幻
- 都市现代类：都市异能、都市日常、都市脑洞、现实题材、黑暗题材、电竞、直播文
- 言情类：古言、宫斗宅斗、青春甜宠、豪门总裁、职场婚恋等
- 特殊题材：规则怪谈、悬疑脑洞、历史古代、知乎短篇、克苏鲁等

#### Step 2：角色骨架与关系冲突
收集信息：
- 主角姓名、欲望、缺陷
- 主角结构（单主角/多主角）
- 感情线配置（无/单女主/多女主）
- 反派分层（小/中/大）与镜像对抗

#### Step 3：金手指与兑现机制
收集信息：
- 金手指类型（可为"无金手指"）
- 名称/系统名、风格、可见度
- 不可逆代价（必须有代价或明确"无+理由"）
- 成长节奏

#### Step 4：世界观与力量规则
收集信息：
- 世界规模（单城/多域/大陆/多界）
- 力量体系类型、势力格局
- 社会阶层与资源分配

#### Step 5：创意约束包（差异化核心）
- 基于题材映射加载反套路库
- 生成 2-3 套创意包（卖点、反套路规则、硬约束等）
- 三问筛选 + 五维评分辅助决策
- 用户选择最终方案

#### Step 6：一致性复述与最终确认
- 输出初始化摘要草案
- 用户确认后执行生成

**执行命令示例**：
```bash
python "${SCRIPTS_DIR}/webnovel.py" init \
  "{project_root}" "{title}" "{genre}" \
  --protagonist-name "{name}" \
  --target-words {words} \
  --target-chapters {chapters} \
  --golden-finger-name "{gf_name}" \
  --golden-finger-type "{gf_type}" \
  ...
```

### 第二步：大纲规划 `/webnovel-plan [卷号]`

**用途**：将总纲细化为卷级规划与章节大纲。

**示例**：
```bash
/webnovel-plan 1
/webnovel-plan 2-3
```

**产出文件**：
- `大纲/第{N}卷-节拍表.md` — 卷级节拍表
- `大纲/第{N}卷-时间线.md` — 时间线表
- `大纲/第{N}卷-详细大纲.md` — 逐章大纲

**工作流程（8 步）**：

| 步骤 | 内容 |
|------|------|
| Step 1 | 加载项目数据（state.json、总纲、设定集） |
| Step 2 | 基于总纲+世界观补齐设定集基线 |
| Step 3 | 选择卷并确认范围 |
| Step 4 | 生成卷节拍表（危机链、中段反转、卷末钩子） |
| Step 4.5 | 生成卷时间线表 |
| Step 5 | 生成卷骨架（Strand 分布、爽点密度、伏笔规划） |
| Step 6 | 分批生成章节大纲 |
| Step 7 | 从卷纲增量补充设定集 |
| Step 8 | 验证 + 保存 + 更新 state |

**每章大纲必须包含**：
- 目标、阻力、代价
- 时间锚点、章内时间跨度、与上章时间差
- 爽点（类型+描述）、Strand 类型
- 反派层级、视角/主角
- 关键实体、本章变化
- 章末未闭合问题、钩子

### 第三步：章节写作 `/webnovel-write [章号]`

**用途**：执行完整章节创作流程（上下文 → 草稿 → 审查 → 润色 → 数据落盘）。

**示例**：
```bash
/webnovel-write 1
/webnovel-write 45
```

**模式**：
- **标准模式**：`/webnovel-write` — 全流程（Step 1→2A→2B→3→4→5→6）
- **快速模式**：`/webnovel-write --fast` — 跳过风格适配（Step 2B）
- **极简模式**：`/webnovel-write --minimal` — 仅 3 个基础审查

**产出文件**：
- `正文/第{NNNN}章-{标题}.md` — 章节正文
- `.webnovel/summaries/ch{NNNN}.md` — 章节摘要
- `index.db` 中的 `review_metrics` — 审查指标
- `.webnovel/state.json` 更新

**工作流程（7 步）**：

| 步骤 | 内容 | 说明 |
|------|------|------|
| Step 0 | 预检 | 校验输入完整性，解析项目路径 |
| Step 1 | Context Agent | 生成创作任务书 + Context Contract + 写作执行包 |
| Step 2A | 正文起草 | 默认 2000-2500 字，禁止占位符 |
| Step 2B | 风格适配 | 仅做表达层转译，`--fast`/`--minimal` 跳过 |
| Step 3 | 审查 | 并行调用 6 维 Checker，必须由子代理执行 |
| Step 4 | 润色 | 修复 critical/high 问题，Anti-AI 终检 |
| Step 5 | Data Agent | 实体提取、状态回写、摘要生成、RAG 索引 |
| Step 6 | Git 备份 | 提交到版本库 |

**六维审查器**：

| Checker | 检查重点 |
|---------|---------|
| consistency-checker | 设定一致性（战力/地点/时间线） |
| continuity-checker | 场景与叙事连贯性 |
| ooc-checker | 人物行为是否偏离人设 |
| reader-pull-checker | 钩子强度、追读力 |
| high-point-checker | 爽点密度与质量 |
| pacing-checker | Strand 比例与断档 |

### 第四步：质量审查 `/webnovel-review [范围]`

**用途**：对历史章节做多维质量审查，生成审查报告。

**示例**：
```bash
/webnovel-review 1-5
/webnovel-review 45
```

**审查深度**：
- **Core（默认）**：consistency / continuity / ooc / reader-pull
- **Full（关键章/用户要求）**：core + high-point + pacing

**产出文件**：
- `审查报告/第{start}-{end}章审查报告.md` — 审查报告
- `index.db` 中的审查指标记录

**工作流程（8 步）**：

| 步骤 | 内容 |
|------|------|
| Step 1 | 加载参考（核心约束） |
| Step 2 | 加载项目状态 |
| Step 3 | 并行调用检查员 |
| Step 4 | 生成审查报告（六维评分 + 优先级） |
| Step 5 | 保存审查指标到 index.db |
| Step 6 | 写回审查记录到 state.json |
| Step 7 | 处理关键问题（询问用户是否立即修复） |
| Step 8 | 收尾 |

## 五、辅助命令

### `/webnovel-query [关键词]`

查询角色、伏笔、节奏、状态等运行时信息。

```bash
/webnovel-query 萧炎        # 查询角色信息
/webnovel-query 伏笔        # 伏笔紧急度分析
/webnovel-query 紧急        # 紧急事项
/webnovel-query 节奏        # Strand 节奏分析
```

### `/webnovel-resume`

任务中断后自动识别断点并恢复。

```bash
/webnovel-resume
```

恢复流程：
1. 检测中断状态
2. 展示恢复选项（已完成/未完成步骤）
3. 用户选择恢复策略（删除重来 / Git 回滚）
4. 执行恢复并继续任务

### `/webnovel-dashboard`

启动只读可视化面板，实时查看项目状态。

```bash
/webnovel-dashboard
```

功能：
- 创作进度与 Strand 节奏分布
- 设定词典（角色/地点/势力等实体）
- 关系图谱
- 章节与大纲内容浏览
- 追读力分析数据

默认访问地址：`http://127.0.0.1:8765`

### `/webnovel-learn [内容]`

从当前会话提取可复用写作模式，写入项目记忆。

```bash
/webnovel-learn "本章的危机钩设计很有效，悬念拉满"
```

产出：`.webnovel/project_memory.json`

## 六、标准创作流程示例

以下是一个完整的创作流程示例：

```
# 1. 初始化项目
/webnovel-init
# 按提示输入书名、题材、主角信息、金手指、世界观等

# 2. 规划第一卷大纲
/webnovel-plan 1
# 确认卷范围，生成节拍表、时间线、详细章纲

# 3. 逐章写作
/webnovel-write 1
/webnovel-write 2
/webnovel-write 3
# ... 依次写完第一卷所有章节

# 4. 定期审查
/webnovel-review 1-10
# 查看审查报告，修复关键问题

# 5. 查询项目状态
/webnovel-query 伏笔
/webnovel-query 节奏

# 6. 规划下一卷
/webnovel-plan 2
# 继续写作...

# 7. 启动可视化面板（可选）
/webnovel-dashboard
```

## 八、常见问题

### Q: 工作区根目录和小说项目根目录有什么区别？

Claude Code 打开的目录是 **工作区根目录**（WORKSPACE_ROOT），它可能包含多个小说项目。每个小说项目有自己的 **项目根目录**（PROJECT_ROOT），由 `/webnovel-init` 按书名创建在工作区下。系统会通过 `.claude/.webnovel-current-project` 指针文件自动追踪当前活跃的小说项目。

### Q: 写作中断了怎么办？

使用 `/webnovel-resume` 命令。系统会自动检测中断位置，展示已完成和未完成的步骤，并提供恢复选项。**不要手动续写半成品内容**，上下文丢失风险高。

### Q: 审查发现了 critical 问题怎么办？

审查报告会按优先级列出问题（高/中/低）。对于 critical 级别问题，系统会询问是否立即修复。建议立即修复后重新运行审查验证。

### Q: 如何切换到另一个小说项目？

修改 `.claude/.webnovel-current-project` 指针文件，指向新的小说项目根目录即可。

### Q: 写作模式如何选择？

- **标准模式**：适合关键章节、需要完整风格适配的场景
- **快速模式**（`--fast`）：适合日常写作，跳过风格适配步骤
- **极简模式**（`--minimal`）：适合赶稿或非关键章节，仅做基础审查
